# วงจรชีวิตของออเดอร์ (Order Lifecycle)

เอกสารนี้คือแหล่งอ้างอิงเดียวของสถานะและการเปลี่ยนสถานะของออเดอร์ รวบรวมจากการตัดสินใจตลอดเซสชัน design ดูศัพท์ที่ใช้ได้ที่ [CONTEXT.md](../CONTEXT.md) และเหตุผลเชิงตัดสินใจที่ [docs/adr/](./adr/)

## แนวคิดหลัก: 2 รางวิ่งคู่กัน

เพราะงานจัดส่งโผล่ให้ไรเดอร์เห็น **ทันทีที่ลูกค้าจ่ายเงิน** (ไม่ใช่หลังร้านรับ) ออเดอร์จึงไม่ใช่สถานะเดียวไหลทีละขั้น แต่มี 2 รางเดินขนานกันแล้วมาบรรจบที่จุด "ไรเดอร์รับอาหารจากร้าน"

```
                         [ลูกค้าจ่ายเงิน]
                               │
            ┌──────────────────┴──────────────────┐
       ราง ร้าน (Merchant)                  ราง ไรเดอร์ (Rider)
            │                                      │
      รอร้านรับ (M0)                        รอไรเดอร์คว้า (R0)
            │                                      │
      ทำอาหาร (M1)                         ไรเดอร์คว้าแล้ว (R1)
            │                                      │
   อาหารเสร็จ-รอรับ (M2)                    ไรเดอร์ถึงร้าน (R2)
            │                                      │
            └──────────────┬───────────────────────┘
                           ▼
               ◆ จุดบรรจบ: ไรเดอร์รับอาหาร ◆
              (ต้องการ M2 "อาหารเสร็จ" และ R2 "ไรเดอร์ถึงร้าน" พร้อมกัน)
                           │
                  ไรเดอร์รับของแล้ว (R3) → กำลังส่ง
                           │
                   ถึงที่อยู่ลูกค้า (R4)
                           │
            ┌──────────────┴──────────────┐
       ลูกค้าให้ OTP                  ส่งไม่ได้ (ลูกค้าไม่อยู่/ไม่รับสาย)
            │                        รอ 10 นาที + โทร 3 ครั้ง
            ▼                              │
   ✅ สำเร็จ (Completed)            ❌ Failed Delivery
                                    (ลูกค้าผิด: ไม่คืนเงิน ·
                                     ไรเดอร์ได้ค่าส่ง · ร้านได้ค่าอาหาร ·
                                     อาหารเป็นของไรเดอร์)
```

### กฎจุดบรรจบ
ไรเดอร์จะ "รับอาหาร" (R2 → R3) ได้ก็ต่อเมื่อ **อาหารเสร็จแล้ว (M2)** เท่านั้น
- ไรเดอร์ถึงร้านก่อนอาหารเสร็จ (R2 แต่ยัง M1) → ไรเดอร์รอที่ร้าน
- อาหารเสร็จก่อนไรเดอร์มา (M2 แต่ยัง R0/R1) → อาหารรอบนเคาน์เตอร์ (นี่คือสถานการณ์ที่นำไปสู่ Delivery Timeout)

## เส้นทางจบก่อนสำเร็จ (Terminal ที่ไม่ใช่ Completed)

| สถานะจบ | เกิดได้ช่วงไหน | ใครกด/ทริกเกอร์ | ผลตามโมเดลความรับผิด (ADR 0002) |
|---|---|---|---|
| **ปฏิเสธโดยร้าน** (Rejected) | M0–M1 และยังไม่ส่งมอบของ | ร้าน (เช่น ของหมด) | ลูกค้าคืนเต็ม · ร้านจ่ายค่าชดเชยไรเดอร์ที่คว้าแล้ว · ร้านโดนลดอันดับ |
| **ยกเลิกโดยลูกค้า** (Cancelled) | เฉพาะตอน **R0** (ยังไม่มีไรเดอร์คว้า) | ลูกค้า | ในหน้าต่างฟรี: คืนเต็ม แพลตฟอร์มกลืนค่าอาหาร · พ้นหน้าต่างฟรี: ยกเลิกฟรีไม่ได้ ต้องรอไรเดอร์หรือ timeout |
| **หมดเวลาไร้ไรเดอร์** (Delivery Timeout) | R0 ค้างเกิน **Y** นาที | ระบบอัตโนมัติ | ลูกค้าคืนเต็ม · แพลตฟอร์มกลืนค่าอาหาร (ไม่มีใครผิด) |
| **ส่งไม่ได้** (Failed Delivery) | R4 (ถึงที่อยู่แล้ว) หลังรอ 10 นาที + โทร 3 ครั้ง | ไรเดอร์ประกาศ | **ลูกค้าผิด** → ลูกค้าไม่คืนเงิน · ไรเดอร์ได้ค่าส่ง · ร้านได้ค่าอาหาร · อาหารเป็นของไรเดอร์ |

## การเปลี่ยนสถานะที่ไม่ใช่ terminal

- **Claim หมดอายุ (R1 → R0):** ไรเดอร์คว้าแล้วไม่คืบหน้าภายใน **Z** นาที หรือกด "ยกเลิกงาน" เอง → งานเด้งกลับเข้าลิสต์ให้คนอื่นคว้า · ถ้าทำบ่อยไรเดอร์โดนพักงาน (Rider Suspension)

## เส้นแบ่งสำคัญ 2 เส้น (ผูกกับการยกเลิกของลูกค้า)

1. **หน้าต่างยกเลิกฟรี (Free Cancellation Window):** เวลาสั้นๆ หลังจ่ายเงิน — ยกเลิกได้ฟรี แพลตฟอร์มกลืนค่าอาหาร
2. **ไรเดอร์คว้างาน (R0 → R1):** เมื่อมีไรเดอร์คว้าแล้ว ลูกค้ายกเลิกไม่ได้อีก (เส้นแบ่งผูกกับ "ไรเดอร์คว้า" ไม่ใช่ "ร้านเริ่มทำ")

## ตัวจับเวลา (parameters)

ค่ากำหนดอยู่ที่ `src/domain/order/timers.ts` (พร้อม predicate แปลง "เวลาที่ผ่านไป → บูลีน" ที่ `transitions.ts` ใช้)

| พารามิเตอร์ | ค่า | ความหมาย |
|---|---|---|
| **หน้าต่างยกเลิกฟรี** | **90 วินาที** | หลังจ่ายเงิน ยกเลิกฟรีได้ภายในเวลานี้ (สั้น กันแพลตฟอร์มกลืนค่าอาหาร) |
| **Y = Delivery Timeout** | **15 นาที** | ไม่มีไรเดอร์คว้าครบ Y → ระบบยกเลิกอัตโนมัติ (คืนเต็ม) |
| **Z = Claim Expiry** | **8 นาที** | ไรเดอร์คว้าแล้วไม่คืบหน้าครบ Z → ปลดงานคืนลิสต์ |
| **ส่งไม่ได้ (Failed Delivery)** | **รอ 10 นาที + โทร 3 ครั้ง** | ครบทั้งเวลาและจำนวนสาย (เงื่อนไข AND) จึงประกาศส่งไม่ได้ |

ขอบเขต: หน้าต่างฟรีนับแบบรวมขอบ (≤90 วิ ยังฟรี) · timeout/claim ครบกำหนดคือทริกเกอร์ (≥)

หน้า Track (เดโม) สาธิต **auto-fire**: นาฬิกาจำลองเดินครบ Y → เรียก `deliveryTimeout` เอง, ครบ Z (นับจากไรเดอร์คว้า) → เรียก `releaseClaim` เอง — predicate ใน `timers.ts` เป็นคนตัดสิน UI แค่ยิง transition
